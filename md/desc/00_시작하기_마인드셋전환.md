# Chapter 0. 시작하기 — 마인드셋 전환 (Java → JS/TS)

> 대상: Java/SpringBoot/MyBatis 20년차
> 목표: "왜 다르게 생겼는지" 부터 이해하고 시작한다.
> 기준: SPECODE 프로젝트 (Next.js 16 + React 19 + TypeScript 6 + Prisma)

---

## 0-1. 큰 그림 한 장 — Spring vs Next.js

| 구분 | Spring + JSP/Thymeleaf | Next.js (우리 프로젝트) |
|---|---|---|
| **언어** | Java | TypeScript (=JavaScript + 타입) |
| **실행 방식** | JVM 위에서 항상 띄워둔 서버 | Node.js + 브라우저, 둘 다 같은 코드를 나눠 실행 |
| **컨트롤러** | `@RestController` 클래스 | `route.ts` 파일 (파일 경로 = URL) |
| **뷰 템플릿** | JSP / Thymeleaf | React 컴포넌트 (`.tsx`) |
| **DB 접근** | MyBatis Mapper XML | **Prisma** (코드로 쿼리, 타입 자동 생성) |
| **DI 컨테이너** | Spring `@Autowired` | 없음. **모듈 import** 로 직접 가져옴 |
| **서비스 계층** | `@Service` 클래스 | `src/lib/*.ts` 의 export 함수들 |
| **세션** | HttpSession | JWT (Authorization 헤더), localStorage |
| **트랜잭션** | `@Transactional` | `prisma.$transaction(async (tx) => {...})` |
| **빌드 산출물** | `.war` / `.jar` | `.next/` 폴더 + Node 프로세스 |

> **핵심**: Next.js 는 *서버 + 클라이언트* 가 **같은 언어, 같은 프로젝트, 같은 import** 안에 공존한다.
> 둘이 어떻게 갈리는지를 이해하는 것이 1순위다.

## 0-2. "한 프로젝트에 서버와 클라이언트가 같이 있다"는 충격

Spring 시절에는 분명했다.

- 서버: Tomcat 위 Java
- 클라이언트: 브라우저 위 JavaScript
- 통신: REST API 또는 JSP 렌더링

**Next.js 는 이 둘을 같은 폴더 안에서 다룬다.**

[src/app/](../../src/app/) 폴더를 열어보면:

```
src/app/
├── api/                    ← 서버 코드 (Tomcat 안 컨트롤러 같은 곳)
│   └── projects/route.ts   ← GET /api/projects 엔드포인트
└── (main)/
    └── projects/page.tsx   ← 브라우저에서 렌더링되는 페이지
```

| 파일 위치 | Spring 비유 | 실행 환경 |
|---|---|---|
| `src/app/api/**/route.ts` | `@RestController` 의 메서드 | **Node.js (서버)** |
| `src/app/**/page.tsx` | JSP / Thymeleaf 뷰 | 1차 렌더는 서버, 이후 **브라우저** |
| `src/lib/*.ts` | Service 계층 (DAO 포함) | 호출하는 쪽에 따라 다름 |

> **규칙 (우리 프로젝트 [.claude/develop/A-NEXTJS-기술규칙.md](../../.claude/develop/A-NEXTJS-기술규칙.md)):**
> - `app/api/` 폴더에서만 DB 접근. UI 코드 절대 금지.
> - `app/[페이지]/` 폴더에서만 UI. DB/Prisma 코드 절대 금지.
> - **혼용은 보안 사고의 99%다.**

## 0-3. Java 와 JavaScript/TypeScript 의 결정적 차이 5가지

### ① 컴파일이 "있다/없다"가 아니라 "타입체크용"이다

- Java: `.java` → `javac` → `.class` (실행 단위)
- TypeScript: `.ts` → `tsc` → `.js` (**타입을 지우기만 함**)

> TS는 런타임에서는 그냥 JS 다. 타입은 *개발 중에만* 살아있다.
> `tsc --noEmit` 이라는 명령이 우리 [package.json](../../package.json) 에 있는 이유 — *체크만 하고 출력 안 함*.

```ts
// Java 의 generics 와 비슷하지만, 런타임에는 흔적도 없다
function getById<T>(id: number): T { ... }
```

### ② "객체"가 자유분방하다

```ts
// 클래스 없이도 객체를 만들 수 있다
const user = { name: "홍길동", age: 30 };

// 타입은 별도로 선언 — 강제는 컴파일 시점에만
type User = { name: string; age: number };
```

Spring 처럼 `class UserDto` 를 매번 만들 필요가 없다. **`type` 으로 모양만 선언**하고 끝.
실제 우리 코드 [src/app/(main)/projects/page.tsx:25-32](../../src/app/(main)/projects/page.tsx#L25-L32) 에서:

```ts
type ProjectItem = {
  projectId:  string;
  name:       string;
  clientName: string | null;
  startDate:  string | null;
  endDate:    string | null;
  myRole:     string;
};
```

이건 클래스가 아니다. **컴파일 후 사라지는** 타입 선언일 뿐이다.

### ③ DI 가 없다 — `import` 가 대신한다

| Spring | Next.js |
|---|---|
| `@Autowired UserService userService;` | `import { findUser } from "@/lib/userService";` |

`@Service`, `@Repository`, `@Component` 같은 어노테이션은 없다.
**모듈을 직접 import** 하면 끝.

우리 코드 [src/app/api/projects/route.ts:6-9](../../src/app/api/projects/route.ts#L6-L9):

```ts
import { NextRequest } from "next/server";
import { prisma } from "@/lib/prisma";        // ← Repository 역할
import { requireAuth } from "@/lib/requireAuth"; // ← Filter/Interceptor 역할
import { apiSuccess, apiError } from "@/lib/apiResponse"; // ← Response 헬퍼
```

`@/` 는 우리 프로젝트의 alias로, `src/` 폴더를 가리킨다 ([tsconfig.json](../../tsconfig.json) 에 정의).

### ④ 스레드가 아니라 "이벤트 루프"다

Java 는 요청마다 스레드 1개를 쓴다.
Node.js 는 **싱글 스레드 + 비동기**로 모든 요청을 처리한다.

→ **모든 I/O 는 `await` 로 대기**한다. (DB, 파일, HTTP 모두)

```ts
// Java MyBatis 스타일 (블로킹)
List<Project> projects = mapper.findAll();

// Next.js + Prisma (비동기, await 필수)
const projects = await prisma.tbPjProject.findMany();
```

`await` 빠뜨리면 **`Promise` 객체가 그대로 변수에 들어간다.** 이게 초보자가 가장 많이 당하는 함정이다.

### ⑤ "null" 이 두 종류다 — `null` 과 `undefined`

- `null`: "값이 명시적으로 비어있다" (Java 의 `null` 과 동일한 느낌)
- `undefined`: "값이 아예 정의되지 않았다" (선언만 하고 할당 안 함)

```ts
let a: string;        // undefined
let b: string = null; // null (옵션에 따라 에러)

// 둘 다 잡고 싶으면 == (==는 "느슨한 비교")
if (value == null) { ... } // null OR undefined 둘 다 true
```

> 우리 프로젝트는 `??` (Nullish Coalescing) 를 즐겨 쓴다.
> `a ?? b` = "a 가 null/undefined 이면 b" (Java 의 `Optional.orElse()` 와 같은 느낌)

[src/app/api/projects/route.ts:45](../../src/app/api/projects/route.ts#L45):
```ts
const aTime = (a.project.mdfcn_dt ?? a.project.creat_dt).getTime();
//             수정일이 null 이면 → 생성일
```

## 0-5. 알아두면 편한 용어집

| 용어 | 한 줄 설명 | Java 비유 |
|---|---|---|
| **JSX / TSX** | JS/TS 안에 HTML 처럼 쓰는 문법 | JSP 의 스크립틀릿 + 태그가 한 파일 안에 |
| **컴포넌트(Component)** | UI 한 조각을 만드는 함수 | JSP 의 커스텀 태그 |
| **Hook** | React 함수 컴포넌트에서 상태/생명주기를 다루는 함수 | (직접 비유 없음. 새 개념) |
| **Props** | 컴포넌트에 넘기는 인자 | 메서드 파라미터 |
| **State** | 컴포넌트가 기억하는 값 | 인스턴스 필드 |
| **Server Component** | 서버에서 한 번만 렌더링되는 컴포넌트 | JSP 와 비슷 |
| **Client Component** | 브라우저에서 동작하는 컴포넌트 | jQuery 시절의 그 자리 |
| **Hydration** | 서버 HTML 위에 JS 가 "살아나는" 과정 | (Java 에 없는 개념) |
| **Mutation** | 데이터 변경 (POST/PUT/DELETE) | Service 의 write 메서드 |
| **Query** | 데이터 조회 (GET) | Service 의 read 메서드 |

## 0-6. 마음가짐

1. **타입에 의존하라.** Java 처럼 IDE 가 빨간 줄 그어주는 게 TS 의 가장 큰 무기다. `any` 를 쓰는 순간 Java 의 `Object` 캐스팅 지옥과 같다.
2. **서버/클라이언트 경계를 항상 의식하라.** 파일 첫 줄의 `"use client"` 유무가 결정한다.
3. **`async/await` 빠뜨리지 마라.** Java 의 `null` 체크보다 더 자주 사고가 난다.
4. **모든 import 는 명시적으로.** `@/lib/...` 경로가 곧 의존성 그래프다.
