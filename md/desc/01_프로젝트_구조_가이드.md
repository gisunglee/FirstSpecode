# SPECODE 프로젝트 구조 가이드

> **대상 독자**: Java/Spring Boot/MyBatis 경험 10년+ / Next.js 처음
> **목적**: 운영자 입장에서 "이 화면은 어디 있고, 어떻게 동작하는가" 를 읽을 수 있게 하는 지도

---

## 0. 한 줄 요약 — Spring Boot와의 매핑

| Spring Boot 세계 | Next.js 세계 (이 프로젝트) | 위치 |
|------------------|----------------------------|------|
| `@RestController` (서버 API) | **API Route** (`route.ts`) | `src/app/api/**` |
| `@Controller` + Thymeleaf/JSP | **Page** (`page.tsx`) | `src/app/(group)/**` |
| `@Service` (비즈 로직) | **lib 헬퍼** (`*.ts`) | `src/lib/**` |
| MyBatis Mapper / JPA Repo | **Prisma Client** (`prisma.tableName.find/create...`) | `src/lib/prisma.ts` 호출 |
| `application.yml` + Filter | **middleware / requireAuth** | `src/lib/requireAuth.ts` |
| `WebMvcConfigurer` 인터셉터 | **requirePermission** (권한 게이트) | `src/lib/requirePermission.ts` |
| Common DTO/Wrapper | **apiSuccess / apiError** | `src/lib/apiResponse.ts` |
| RestTemplate / OpenFeign | **apiFetch / authFetch** | `src/lib/apiFetch.ts`, `src/lib/authFetch.ts` |
| Bean Validation (`@Valid`) | **Zod 스키마** | 각 route 내부 또는 `lib/` |
| 타임리프 layout decorator | **`layout.tsx`** (중첩 레이아웃) | `src/app/**/layout.tsx` |
| 세션 관리(JSESSIONID) | **JWT(AT) + sessionStorage** | `src/lib/auth.ts` |
| Spring Security Role | **permissions 매트릭스** | `src/lib/permissions.ts` |

> **핵심 차이 1**: Next.js는 **파일 = URL**. `src/app/projects/[id]/screens/page.tsx` 라는 파일 위치 자체가 URL `/projects/:id/screens` 다.
> Spring처럼 `@RequestMapping("/screens")` 같은 매핑 어노테이션이 없다. **폴더 구조가 라우팅 테이블**이다.
>
> **핵심 차이 2**: 한 프로세스 안에서 **서버 코드(API Route)와 클라이언트 코드(Page)가 같은 저장소에 공존**한다. 구분 기준은 폴더(`api/` vs 그 외) + 파일명(`route.ts` vs `page.tsx`) + `"use client"` 선언.

### 2-1. `src/app/` — 페이지·API 라우팅의 절대 규칙

**Next.js App Router는 "폴더 = URL, 파일명에 의미가 있다"** 가 전부입니다.

```
src/app/
├── layout.tsx           ← 모든 페이지의 최상위 레이아웃 (HTML <html><body> 정의)
├── page.tsx             ← 루트 "/" 페이지 (현재는 단순 리다이렉트)
├── globals.css          ← 전역 CSS
│
├── (auth)/              ← URL에 영향 X (괄호 그룹). 로그인/회원가입 묶음
│   ├── layout.tsx       ← 이 그룹만의 레이아웃 (로그인용 단순 레이아웃)
│   ├── auth/login/page.tsx       → /auth/login
│   ├── auth/register/page.tsx    → /auth/register
│   ├── auth/password/page.tsx    → /auth/password
│   └── invite/accept/page.tsx    → /invite/accept
│
├── (main)/              ← URL에 영향 X. GNB+LNB+StatusBar 들어가는 워크스페이스 묶음
│   ├── layout.tsx       ← MainLayout (GNB/LNB 포함)
│   ├── dashboard/page.tsx        → /dashboard
│   ├── settings/account/page.tsx → /settings/account
│   ├── admin/...                 → /admin/** (시스템 관리자 전용)
│   └── projects/
│       ├── page.tsx              → /projects (프로젝트 목록)
│       └── [id]/                 ← [폴더] = 동적 파라미터 (= Spring @PathVariable)
│           ├── settings/page.tsx        → /projects/:id/settings
│           ├── members/page.tsx         → /projects/:id/members
│           ├── tasks/page.tsx           → /projects/:id/tasks
│           ├── requirements/
│           │   ├── page.tsx             → /projects/:id/requirements
│           │   └── [reqId]/page.tsx     → /projects/:id/requirements/:reqId
│           ├── screens/                 → 화면 (Screen)
│           ├── areas/                   → 영역
│           ├── functions/               → 기능
│           ├── ai-tasks/                → AI 태스크
│           └── ... (총 25개 엔티티 폴더)
│
└── api/                 ← ★ 서버 전용 (Spring의 RestController 영역)
    ├── auth/login/route.ts        → POST /api/auth/login
    ├── auth/register/route.ts     → POST /api/auth/register
    ├── projects/route.ts          → GET/POST /api/projects
    ├── projects/[id]/route.ts     → GET/PUT/DELETE /api/projects/:id
    ├── projects/[id]/screens/route.ts            → GET/POST
    ├── projects/[id]/screens/[screenId]/route.ts → GET/PUT/DELETE
    ├── admin/...                  → 시스템 관리자 API
    ├── mcp/...                    → MCP 서버용 API
    └── worker/...                 → 백그라운드 워커용 API
```

#### 파일명·폴더명 규칙 (외워야 할 것 5개)

| 이름 | 의미 | Spring 비유 |
|------|------|-------------|
| `page.tsx` | URL에 매핑되는 **화면** 컴포넌트 | View (JSP/Thymeleaf) |
| `route.ts` | URL에 매핑되는 **API 엔드포인트** | RestController 메서드 |
| `layout.tsx` | 자식 라우트를 감싸는 **공통 레이아웃** | 데코레이터 / 인터셉터 + view |
| `[xxx]` | **동적 세그먼트** (폴더명) | `@PathVariable("xxx")` |
| `(xxx)` | **그룹 폴더** — URL에는 영향 X, 레이아웃 분리용 | 패키지 정도 |

> **레이아웃은 자동으로 중첩됩니다.**
> `/projects/123/screens` 페이지를 그리면:
> `app/layout.tsx` → `(main)/layout.tsx (MainLayout=GNB+LNB)` → `projects/[id]/screens/page.tsx`
> 이 순서로 React가 트리를 구성. Spring으로 치면 SiteMesh/Tiles 데코레이터가 자동 적용되는 셈.

#### 페이지(`page.tsx`) vs API(`route.ts`) — 절대 혼용 금지

```
src/app/api/**       ← 서버에서만 실행. Prisma 호출, 비밀키 사용 OK
src/app/(main)/**    ← 브라우저로 전송됨. DB 직접 접근 금지, 환경변수 노출 금지
```

> Java로 치면 **`api/` = `*.java` (서버 컴파일), `(main)/` = `*.jsp` (브라우저 출력)** 와 같다고 생각하면 됩니다. 두 코드가 같은 저장소에 있어 헷갈리지만, 절대 같은 파일에 섞지 않습니다.

### 2-2. 라우팅 동작 — 사용자 클릭 → DB까지 한 사이클

예: 사용자가 화면 목록 페이지에서 "화면 추가" 를 누른다.

```
[브라우저]
1. 사용자: /projects/abc/screens 진입
   ↓
2. Next.js: 폴더 매칭 → src/app/(main)/projects/[id]/screens/page.tsx 렌더
   ↓
3. ScreensPage.tsx 안의 useQuery 가
      authFetch("/api/projects/abc/screens")  호출
   ↓
[Next.js 서버 (같은 프로세스)]
4. 라우팅: src/app/api/projects/[id]/screens/route.ts 의 GET 함수 실행
   ↓
5. requireAuth(request)        ← JWT 검증 (= Spring Security Filter)
6. requirePermission(...)      ← 역할/직무 권한 검증 (= @PreAuthorize)
7. prisma.tbDsScreen.findMany  ← DB 조회 (= MyBatis selectList)
8. apiSuccess({ data: rows })  ← 일관된 응답 포맷으로 직렬화
   ↓
[브라우저]
9. TanStack Query가 응답 캐시 → 컴포넌트 자동 리렌더
10. 화면에 목록 표 출력
```

핵심 요점:
- **3번의 `apiFetch`/`authFetch`** = Spring의 `RestTemplate`. 모든 API 호출은 이 래퍼만 사용.
- **5/6번 게이트** = 모든 API의 첫 줄. 빠지면 보안 사고.
- **7번 prisma** = `prisma.테이블명.동작(...)`. SQL 직접 작성 안 함 (parameterized 자동).

#### 컴포넌트 호출 흐름 예 (Spring으로 치면 fragment include)

```tsx
// src/app/(main)/projects/[id]/screens/page.tsx (요지)
import AiTaskDetailDialog from "@/components/ui/AiTaskDetailDialog";

export default function ScreensPage() {
  const [openAiTaskId, setOpenAiTaskId] = useState<string|null>(null);

  return (
    <>
      <table>... 목록 ...</table>

      {/* 모달은 페이지 JSX 끝에 두고 open으로 제어 */}
      {openAiTaskId && (
        <AiTaskDetailDialog
          taskId={openAiTaskId}
          onClose={() => setOpenAiTaskId(null)}
        />
      )}
    </>
  );
}
```

`@/components/...` 의 `@` 는 `src/` 의 별칭 (tsconfig.json의 paths 설정). Spring의 `import com.company.app...` 에 해당.

#### `lib/`의 황금 패턴 — API Route 첫 5줄 외우기

거의 모든 API route가 이 형태로 시작합니다.

```ts
export async function GET(request: NextRequest, { params }: RouteParams) {
  const { id: projectId } = await params;          // 1. 파라미터 파싱 (Next 16: await 필수)

  const auth = await requireAuth(request);          // 2. JWT/API키 인증
  if (auth instanceof Response) return auth;        //    실패면 401/403 즉시 반환

  const gate = await requirePermission(             // 3. 역할/직무 권한 체크
    request, projectId, "content.read"
  );
  if (gate instanceof Response) return gate;

  try {
    const rows = await prisma.tbXxx.findMany({...}); // 4. DB 조회
    return apiSuccess({ data: rows });               // 5. 일관 포맷 응답
  } catch (e) {
    return apiError("INTERNAL_ERROR", "...", 500);
  }
}
```

> Spring으로 치면 `@RestController` + `@PreAuthorize` + `@Valid` + `try/catch + ResponseEntity` 가 모두 합쳐진 한 함수입니다. 어노테이션 대신 첫 줄 함수 호출로 처리.

## 3. 데이터 계층 — Prisma + DB

```
prisma/
├── schema.prisma   ← 1069줄. 모든 테이블 정의 + 관계 + 인덱스
└── sql/            ← 수동 마이그레이션 (컬럼 추가 등 schema.prisma로 안 되는 것)
```

### Prisma vs MyBatis

| 항목 | MyBatis | Prisma |
|------|---------|--------|
| 테이블 정의 | XML mapper / DDL | `schema.prisma` 한 파일 |
| 쿼리 작성 | XML 또는 어노테이션 | TypeScript 메서드 체인 (`prisma.user.findMany`) |
| 결과 타입 | DTO 직접 정의 | **자동 생성** (테이블에서 추론) |
| 동적 쿼리 | `<if>`, `<foreach>` | JS 객체 (`where: { ... }`) |
| 페이징 | 직접 구현 | `skip`, `take` |
| 트랜잭션 | `@Transactional` | `prisma.$transaction([...])` |

호출 형태:
```ts
// SELECT * FROM tb_ds_screen WHERE prjct_id = ? AND use_yn = 'Y' LIMIT 20
const rows = await prisma.tbDsScreen.findMany({
  where: { prjct_id: projectId, use_yn: "Y" },
  take: 20,
  orderBy: { sort_order: "asc" },
  include: { area: true },     // JOIN
});
```

> **Prisma는 항상 parameterized query** 라 SQL injection 자체가 거의 불가. 단, 비즈니스 검증(빈 값, 음수, 길이 등)은 직접 해야 합니다 (Zod 또는 수동).

## 4. 인증·권한 — 보안 레이어 한 장 정리

```
[브라우저]                         [Next.js 서버]
  ─ 로그인 폼  ──────────────────► /api/auth/login (route.ts)
                                    └── auth.ts: bcrypt.compare → JWT 발급
  ◄────────────── { accessToken } ──┘
  sessionStorage 저장
                                  
  ─ 모든 API 호출 ────────────────► /api/...
    (authFetch가 Authorization      └── requireAuth(req)        ← JWT 검증
     헤더 자동 첨부)                 └── requirePermission(...) ← 역할 체크
                                     └── prisma.xxx              ← 비즈 로직
```

- **JWT 액세스 토큰**: 30분 유효, `sessionStorage.access_token`
- **리프레시 토큰**: 10일, DB 보관 (해시)
- **MCP API 키 (`spk_` prefix)**: AI 도구가 사용. JWT 대신 키로 인증 (`requireAuth`가 자동 분기)
- **권한 매트릭스 (`lib/permissions.ts`)**: 역할(OWNER/ADMIN/MEMBER/VIEWER) × 직무(PM/PL/DEV/...) 두 축 OR 결합
- **시스템 관리자 (SUPER_ADMIN)**: 별도 컬럼. 고객 프로젝트 "지원 세션" 진입 시 **읽기 전용**으로만 동작 (write 권한은 자동 차단)
- **CLAUDE.md §9**: 데이터 내보내기는 `content.read` 가 아닌 별도 `content.export` 권한으로 분리해야 지원 세션에서도 안전

> Spring의 `SecurityFilterChain` + `@PreAuthorize` 조합을 **함수 호출 두 줄**로 표현했다고 보면 됩니다.

## 6. MCP 서버 (HTTP) — Claude(AI)와 시스템을 잇는 게이트웨이

별도 프로세스 없이 **Next.js 라우트 안에서 직접 처리**합니다.

```
src/
├── app/api/mcp/route.ts          ← MCP HTTP 엔드포인트 (POST /api/mcp)
└── lib/mcp/
    ├── register-tools.ts          ← AI에게 노출할 "도구" 정의 (= 서비스 외부 인터페이스)
    ├── api-client.ts              ← 본 Next.js 서버의 API를 호출하는 래퍼
    └── tools/, lib/
```

핵심: **MCP 라우트는 본 앱의 API를 다시 호출** 합니다. AI에게 "도구"를 노출하기 위한 얇은 게이트웨이.
- 인터페이스 변경 시 `src/lib/mcp/register-tools.ts` 한 곳만 수정 (CLAUDE.md "MCP 동기화 규칙" 참고)
- 인증: `Authorization: Bearer spk_...` 헤더 (`tb_mcp_key` 의 `purpose='MCP'` 키)

> 과거에는 `mcp-server/` 별도 Node 프로세스(stdio)를 두었으나 폐기되었습니다. HTTP MCP 한 채널로 일원화.

> Spring으로 치면 "공개 OpenAPI를 자동으로 OpenFeign 클라이언트로 만들어 외부 BFF로 노출" 정도. 다만 호출 주체는 사람이 아니라 LLM.

## 7. 운영 시 자주 보게 될 시나리오 — "어디부터 보면 되나"

### 시나리오 A: "특정 페이지가 500 에러"
1. 브라우저 DevTools → Network 에서 실패한 `/api/...` URL 확인
2. URL → `src/app/api/...` 폴더 따라 들어가서 `route.ts` 열기
3. 그 함수의 try/catch 안 prisma 호출과 권한 체크 흐름 확인
4. 서버 콘솔(`npm run dev` 띄운 터미널) 의 스택트레이스 확인

### 시나리오 B: "이 버튼이 뭐 하는 거지?"
1. 페이지 파일(`page.tsx`) 의 파일 상단 주석부터 읽기 (역할/PID/FID 명시됨)
2. 버튼의 `onClick` 추적 → `useMutation` / 다이얼로그 컴포넌트로 이동
3. 다이얼로그는 **`components/ui/*Dialog.tsx`**

### 시나리오 C: "권한이 막혀서 접근이 안 된대요"
1. `src/lib/permissions.ts` 의 `PERMISSIONS` 매트릭스 확인
2. 해당 페이지/API에서 `requirePermission(req, projectId, "...")` 의 키가 무엇인지 확인
3. 사용자 역할(role) + 직무(job) 가 그 키의 허용 목록에 있는지 검증

### 시나리오 D: "DB 컬럼이 추가됐어요"
1. `prisma/schema.prisma` 수정
2. `npm run db:push` (개발) 또는 `prisma/sql/` 에 마이그레이션 작성 (운영)
3. `npm run db:generate` (Prisma Client 재생성)
4. 영향받는 API route / 페이지 코드 수정

### 시나리오 E: "팝업을 새로 추가하고 싶어요"
1. `src/components/ui/MyDialog.tsx` 신규 생성 (`"use client"` 필수)
2. 페이지에서 `useState` 로 열림 상태 관리
3. JSX 끝에 `{open && <MyDialog ... onClose={() => setOpen(false)} />}` 식
