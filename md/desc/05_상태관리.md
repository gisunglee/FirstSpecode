# Chapter 5. 상태 관리 — useState / Zustand / TanStack Query 의 역할 분담

> **이 챕터의 정체**: "어디에 어떤 데이터를 보관하느냐" 의 결정 기준.
> Java 비유: HttpSession / Application Scope / Request Scope / DB 의 트레이드오프와 같은 문제.

---

## 5-1. 상태의 4가지 위치

| 위치 | 도구 | 비유 (Spring) | 보관 기간 |
|---|---|---|---|
| **컴포넌트 내부** | `useState` | 메서드 지역 변수 | 컴포넌트 살아있는 동안 |
| **앱 전역** | Zustand | ServletContext / Singleton Bean | 페이지 새로고침까지 (또는 영속화) |
| **서버 응답 캐시** | TanStack Query | EhCache / Redis | staleTime 동안 |
| **URL** | `useSearchParams` | request param | URL 그대로 |

> 한 가지 도구로 다 처리하면 안 된다. **데이터의 성격에 맞게** 나눠 보관.

---

## 5-2. 결정 기준 — 어떤 데이터를 어디에?

### A. 이 데이터, 다른 컴포넌트도 봐야 하나?
- **아니오** → `useState`
- **예** → Zustand 또는 props 전달

### B. 이 데이터, 서버에서 받아오나?
- **예** → TanStack Query
- **아니오** → useState/Zustand

### C. URL 에 담아 공유 가능한가? (예: 검색어, 필터)
- **예** → URL 쿼리스트링이 진리. `useSearchParams` 로 읽기.
- **아니오** → useState/Zustand

## 5-3. `useState` — 컴포넌트 지역 변수

이미 [Chapter 2](./02_React_기본.md) 에서 봤으니 짧게.

```tsx
const [open, setOpen] = useState(false);
const [text, setText] = useState("");
```

**용도**:
- 폼 입력값
- 모달 열림/닫힘
- 토글
- "이 컴포넌트만 알면 되는" 모든 것

> Java 의 메서드 지역 변수처럼 **떠나면 사라짐**. 그래서 가장 안전하고 단순.

### 우리 프로젝트 — [src/app/(main)/projects/page.tsx:75-79](../../src/app/(main)/projects/page.tsx#L75-L79)
```tsx
function CreateProjectDialog({ ... }) {
  const [name,        setName]        = useState("");
  const [description, setDescription] = useState("");
  const [startDate,   setStartDate]   = useState("");
  const [endDate,     setEndDate]     = useState("");
  const [clientName,  setClientName]  = useState("");
  // ...
}
```

폼 입력은 무조건 `useState`. 다이얼로그 닫히면 자동 사라짐.

## 5-4. Zustand — 전역 상태 (라이브러리)

### 왜 Context 가 아니라 Zustand 인가?
React 에는 Context API 가 있지만, 큰 앱에서는 다음 문제가 있다:
- Context 가 바뀌면 그 안의 모든 구독자 컴포넌트가 리렌더
- 보일러플레이트 많음 (Provider, Consumer, ...)

**Zustand 는 단순하고 성능 좋다.** SPECODE 의 선택.

### 기본 구조 — [src/store/appStore.ts](../../src/store/appStore.ts)
```ts
import { create } from "zustand";

type AppState = {
  currentProjectId: string | null;
  theme: "dark" | "light" | "dark-purple";
  sidebarCollapsed: boolean;
};

type AppActions = {
  setCurrentProjectId: (id: string | null) => void;
  setTheme: (theme: AppState["theme"]) => void;
  toggleSidebar: () => void;
};

export const useAppStore = create<AppState & AppActions>((set, get) => ({
  // 초기값
  currentProjectId: null,
  theme: "dark",
  sidebarCollapsed: false,

  // 액션
  setCurrentProjectId: (id) => set({ currentProjectId: id }),
  setTheme: (theme) => set({ theme }),
  toggleSidebar: () => set((s) => ({ sidebarCollapsed: !s.sidebarCollapsed })),
}));
```

### 사용
```tsx
function GNB() {
  // 필요한 것만 꺼냄 — 다른 필드 변경 시 리렌더 안 함
  const projectId = useAppStore((s) => s.currentProjectId);
  const setProjectId = useAppStore((s) => s.setCurrentProjectId);

  // 또는 한 번에
  const { theme, setTheme } = useAppStore();

  return (
    <div>
      <button onClick={() => setProjectId("xyz")}>전환</button>
    </div>
  );
}
```

> 사용법이 React 의 `useState` 와 거의 같다. 그게 Zustand 의 장점.

## 5-5. Zustand 의 영속화 (`persist`)

새로고침 후에도 값 유지하려면? `persist` 미들웨어.

### 우리 프로젝트 — [src/store/appStore.ts:58-104](../../src/store/appStore.ts#L58-L104)
```ts
import { persist } from "zustand/middleware";

export const useAppStore = create<AppState & AppActions>()(
  persist(
    (set, get) => ({ ... }),
    {
      name: "specode-app-state",          // localStorage 키
      // 어떤 필드만 영속화할지 선택
      partialize: (state) => ({
        theme: state.theme,
        sidebarCollapsed: state.sidebarCollapsed,
      }),
    }
  )
);
```

> Spring 의 `HttpSession` 과 발상이 비슷하지만, **클라이언트의 localStorage** 에 저장.
> 서버는 모름. **민감한 값(토큰 등)은 절대 여기 넣지 말 것.**

### 영속화 vs 비영속화 결정
| 데이터 | persist? | 이유 |
|---|---|---|
| 테마 (dark/light) | ✅ 함 | 새로고침 후에도 같은 테마 |
| 사이드바 접힘 | ✅ 함 | UX 일관성 |
| 현재 프로젝트 ID | ❌ 안함 | 새로고침 시 재선택 유도 (의도적) |
| 내 담당 모드 | ❌ 안함 | **DB 가 원천**. 서버 프로필이 진리 |

> **저장의 책임이 누구에게 있는지** 가 결정 기준. 우리 프로젝트의 주석이 그 의도를 잘 보여준다.

## 5-6. TanStack Query 도 상태다

[Chapter 4](./04_TanStackQuery_데이터페칭.md) 에서 봤듯, `useQuery` 의 결과도 본질은 "캐시된 상태".

### 의도적 분리
- **서버에서 받은 데이터** → TanStack Query
- **로컬에서 만든 UI 상태** → useState / Zustand

### ❌ 흔한 실수 — 서버 데이터를 useState 에 복사
```tsx
const { data } = useQuery({ queryKey: ["projects"], queryFn: fetchProjects });
const [projects, setProjects] = useState([]);

useEffect(() => {
  if (data) setProjects(data);   // ← 동기화 노력
}, [data]);
```

이러면 두 곳에 데이터가 살게 되어 **싱크 지옥**. 그냥 `data` 를 직접 쓰면 됨.

```tsx
// ✅ 그냥 useQuery 결과를 바로 사용
const { data } = useQuery(...);
const items = data?.items ?? [];
```

## 5-7. URL 도 상태다 — `useSearchParams`

페이지를 공유 가능하게 만드는 가장 좋은 방법. **검색어, 필터, 페이지 번호** 는 URL 에 두는 것이 정석.

```tsx
"use client";
import { Suspense } from "react";
import { useSearchParams, useRouter, usePathname } from "next/navigation";

export default function Page() {
  return (
    <Suspense fallback={null}>
      <Inner />
    </Suspense>
  );
}

function Inner() {
  const router = useRouter();
  const pathname = usePathname();
  const params = useSearchParams();

  // 읽기
  const search = params.get("q") ?? "";
  const page = Number(params.get("page") ?? "1");

  // 쓰기 — URL 변경
  function setSearch(q: string) {
    const next = new URLSearchParams(params);
    next.set("q", q);
    router.push(`${pathname}?${next.toString()}`);
  }

  return <input value={search} onChange={(e) => setSearch(e.target.value)} />;
}
```

> Spring 의 `@RequestParam` 과 같은 의도. 다만 React 에서는 클라이언트 측 라우팅이라
> 페이지 새로고침 없이 URL 만 바뀐다.

## 5-8. Context — 알아둘 것 (안 쓰지만 만남)

```tsx
// 특정 트리에만 적용되는 의존성 주입 비슷한 것
const ThemeContext = createContext<"dark" | "light">("dark");

function App() {
  return (
    <ThemeContext.Provider value="dark">
      <Toolbar />
    </ThemeContext.Provider>
  );
}

function Toolbar() {
  const theme = useContext(ThemeContext);    // "dark"
  return <div>{theme}</div>;
}
```

> Spring 의 `@Scope("request")` 빈 + DI 같은 그림.
> **우리 프로젝트는 거의 안 쓴다.** Zustand 가 전역, useState 가 지역, props 가 단계 전달이라 Context 가 필요한 자리가 적음.

## 5-9. 의사결정 흐름도

```
새 데이터가 필요하다
        │
        ▼
   서버에서 받나?  ─ YES ─→  useQuery (TanStack Query)
        │
        NO
        │
        ▼
   여러 컴포넌트가 공유? ─ YES ─→  Zustand
        │                          (영속 필요? → persist 미들웨어)
        NO
        │
        ▼
   URL 에 노출 가능? ─ YES ─→  useSearchParams
        │
        NO
        │
        ▼
        useState
```

## 5-11. 안티패턴 4가지

| 안티패턴 | 왜 나쁜가 |
|---|---|
| 모든 걸 Zustand 에 넣기 | 영향 범위 너무 넓어짐. 작은 컴포넌트 변경에 전체 영향 추적 어려움 |
| 서버 데이터를 useState 에 복사 | 동기화 지옥. useQuery 만 써라 |
| Zustand 에 토큰 넣기 | 다른 탭과 공유돼서 보안 약화. sessionStorage/localStorage 직접 쓰는 게 더 명시적 |
| URL 에 들어가야 할 걸 useState 에 | 새로고침/공유 시 사라짐 |
