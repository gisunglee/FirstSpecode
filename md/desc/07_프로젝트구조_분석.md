# Chapter 7. 우리 프로젝트(SPECODE) 구조 분석

> **이 챕터의 정체**: 지금까지 배운 모든 개념이 어디에 어떻게 배치되어 있는지 **실제 폴더 트리 한 줄씩** 짚는다.
> Java 비유: Spring 프로젝트의 `src/main/java/`, `src/main/resources/`, `webapp/` 매핑표 같은 것.

---

## 7-1. 최상위 폴더

```
FirstSpecode/
├── src/                    ← 우리가 매일 보는 곳
├── prisma/                 ← DB 스키마 (Chapter 6)
├── public/                 ← 정적 파일 (이미지 등)
├── design/                 ← 디자인 시스템 (CSS 토큰)
├── md/                     ← 기획/PRD/이 교재가 있는 곳
├── scripts/                ← 일회성 스크립트 (DDL 파서 등)
├── package.json            ← 의존성 + npm scripts
├── tsconfig.json           ← TS 설정 (alias `@/` 등)
├── next.config.ts          ← Next.js 설정
└── .claude/                ← Claude Code 작업 규칙
```

## 7-2. `src/` 안의 7개 핵심 폴더

```
src/
├── app/         ← Next.js 의 라우팅. 페이지/API 가 모두 여기.
├── components/  ← 재사용 UI 컴포넌트
├── constants/   ← 상수 (자주 안 봄)
├── hooks/       ← 커스텀 React Hook
├── lib/         ← 비즈니스 로직 + 유틸 (Spring 의 service + util)
├── providers/   ← 전역 Provider (TanStack Query 등)
├── store/       ← Zustand 전역 상태
├── styles/      ← 글로벌 CSS
└── types/       ← 공유 타입
```

각각이 어떤 역할인지 **Spring 프로젝트와 매핑** 해보자.

| 우리 폴더 | Spring 비유 |
|---|---|
| `src/app/api/**` | `@RestController` 들 |
| `src/app/**/page.tsx` | JSP / Thymeleaf 화면 |
| `src/lib/**` | `@Service`, `util/` |
| `src/components/**` | (Java 에 없음. 재사용 UI) |
| `src/hooks/**` | (Java 에 없음. 재사용 동작) |
| `src/store/**` | `@SessionScope` Bean 비슷 |
| `src/providers/**` | `@Configuration` 비슷 |
| `prisma/schema.prisma` | DDL + 도메인 클래스 |

## 7-3. `src/app/` — 라우팅의 전부

```
src/app/
├── layout.tsx                      ← 모든 페이지 최상위 (Provider 등)
├── page.tsx                        ← / (랜딩)
├── globals.css                     ← 전역 CSS
│
├── (auth)/                         ← 라우트 그룹 (URL 영향 없음)
│   └── ... 로그인, 회원가입 페이지
│
├── (main)/                         ← 라우트 그룹 (GNB+LNB 적용)
│   ├── layout.tsx                  ← (main) 그룹 공통 레이아웃
│   ├── dashboard/page.tsx          ← /dashboard
│   ├── projects/page.tsx           ← /projects
│   ├── projects/[id]/...           ← /projects/{id}/*
│   └── admin/...                   ← /admin/*
│
└── api/                            ← 서버 전용
    ├── projects/route.ts           ← /api/projects
    ├── projects/[id]/route.ts      ← /api/projects/{id}
    ├── projects/[id]/tasks/...     ← /api/projects/{id}/tasks/*
    ├── auth/...                    ← /api/auth/*
    ├── mcp/...                     ← MCP 도구 endpoint
    └── worker/...                  ← 백그라운드 워커 endpoint
```

### 그룹 폴더 `()` 의 의미
- `(auth)`, `(main)` → URL 에 영향 안 줌. **레이아웃 공유 목적**.
- `/auth/login` 이 아니라 `/login` 이 됨.
- 우리 코드에서는 [src/app/(auth)/](../../src/app/(auth)/) 와 [src/app/(main)/](../../src/app/(main)/) 로 두 종류의 레이아웃을 분리.

### `[id]` 폴더의 의미 (Chapter 3 복습)
- 동적 세그먼트
- `params.id` 로 받음 (Promise 라 await 필수)

```ts
import { requireAuth }             from "@/lib/requireAuth";
import { apiSuccess, apiError }    from "@/lib/apiResponse";
import { requirePermission }       from "@/lib/requirePermission";  // 권한 필요 시
```

### `ui/` vs `common/` 의 차이
- `ui/` = "버튼, 다이얼로그 같은 **재사용 원자 부품**"
- `common/` = "**우리 프로젝트만의** 비즈니스 패턴 (예: ConfirmDialog 는 모달 + 표준 메시지)"

### `layout/MainLayout.tsx` — 가장 자주 만지는 레이아웃
- GNB (상단 바)
- LNB (좌측 사이드바)
- StatusBar (하단)
- children (실제 페이지 내용)

[src/app/(main)/layout.tsx](../../src/app/(main)/layout.tsx) 가 이걸 wrap 한다.

## 7-7. `src/hooks/` — 커스텀 Hook

```
src/hooks/
├── useEscapeKey.ts                  ← ESC 키로 모달 닫기
├── useGlobalSearchShortcut.ts       ← Ctrl+K 단축키
├── useMyRole.ts                     ← 현재 프로젝트의 내 역할 조회
├── useTripleClick.ts                ← 트리플 클릭 감지
└── useTripleClickSidebarToggle.ts   ← 트리플 클릭 → 사이드바
```

### 커스텀 Hook 이란?
"**`use` 로 시작하는 함수가, 내부에서 다른 hook 을 호출**" 하는 것. 그게 끝.

```tsx
// 예: useEscapeKey
export function useEscapeKey(onEscape: () => void) {
  useEffect(() => {
    const handler = (e: KeyboardEvent) => {
      if (e.key === "Escape") onEscape();
    };
    document.addEventListener("keydown", handler);
    return () => document.removeEventListener("keydown", handler);
  }, [onEscape]);
}

// 사용
function MyDialog({ onClose }: { onClose: () => void }) {
  useEscapeKey(onClose);
  return <div>...</div>;
}
```

> Java 에는 비유가 약하다. **재사용 가능한 동작 묶음**, AOP 의 어드바이스에 가장 가깝다.

## 7-13. `package.json` 의 npm scripts

```json
{
  "scripts": {
    "dev":          "next dev",                     ← 개발 서버 (포트 3001)
    "typecheck":    "tsc --noEmit",                 ← 타입만 체크
    "build":        "tsc --noEmit && next build",   ← 타입 체크 + 빌드
    "start":        "next start",                   ← 프로덕션 모드
    "db:push":      "...prisma db push",            ← schema → DB 반영
    "db:studio":    "...prisma studio",             ← DB GUI 도구
    "db:generate":  "...prisma generate"            ← Client 재생성
  }
}
```

| 명령 | 언제 |
|---|---|
| `npm run dev` | 개발 시작할 때 |
| `npm run typecheck` | 타입 에러만 빨리 확인 |
| `npm run build` | 배포 직전 풀 검증 |
| `npm run db:push` | schema.prisma 수정 후 DB 반영 |
| `npm run db:generate` | schema.prisma 수정 후 TS 타입 재생성 |

## 7-14. 의존성 그래프 — 새 API 만들 때 import 흐름

```
src/app/api/projects/route.ts
       │
       ├─ import { prisma }       from "@/lib/prisma"
       ├─ import { requireAuth }  from "@/lib/requireAuth"
       │       │
       │       └─ import { verifyAccessToken, hashApiKey } from "@/lib/auth"
       │       └─ import { prisma }                         from "@/lib/prisma"
       │
       └─ import { apiSuccess, apiError } from "@/lib/apiResponse"
```

> 모든 API 가 이런 형태. **Spring 의 `@Autowired` 흐름과 동일**, 단지 import 로 명시될 뿐.

## 7-15. 페이지 의존성 흐름

```
src/app/(main)/projects/page.tsx
       │
       ├─ "use client"            ← 클라이언트 컴포넌트
       ├─ import { useQuery, useMutation } from "@tanstack/react-query"
       ├─ import { authFetch }    from "@/lib/authFetch"
       ├─ import { useAppStore }  from "@/store/appStore"
       └─ import { toast }        from "sonner"
                                       ↑
                                   /api/projects 호출 (서버) → DB
```

페이지 → 라이브러리 → API → DB 의 흐름.
**페이지가 직접 DB 접근하는 일은 없다** ([.claude/develop/A-NEXTJS-기술규칙.md](../../.claude/develop/A-NEXTJS-기술규칙.md) 1번 규칙).
