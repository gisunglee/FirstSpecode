# Chapter 8. 실전 코드 읽기 가이드

> **이 챕터의 정체**: 처음 보는 파일을 만났을 때, **어떤 순서로** 어디를 보면 빨리 파악할 수 있는지.
> Java 시절의 "Service → DAO → Mapper.xml" 의 순서 같은 것을 우리 프로젝트로 매핑.

---

## 8-1. 첫 30초 — 무슨 종류의 파일인가?

새 파일을 열면 **첫 5줄**을 보고 결정한다.

| 첫 5줄에 있는 것 | 파일 종류 | 어떻게 읽을지 |
|---|---|---|
| `"use client";` | 클라이언트 컴포넌트 | UI 위주, useState/useQuery 따라가기 |
| `export async function GET(...)` | API route | 인증 → 검증 → DB → 응답 순서로 |
| `export default async function Page` (no `"use client"`) | 서버 컴포넌트 | DB 직접 접근 가능 |
| `import { create } from "zustand"` | Zustand store | 상태 + 액션 |
| `model Foo {` | Prisma 스키마 | DB 테이블 |
| `export function useFoo()` | 커스텀 hook | 재사용 가능한 동작 |

## 8-2. API Route 읽는 순서 — `route.ts`

### 표준 구조 (우리 프로젝트의 컨벤션)
```ts
// ① 헤더 주석 — 무엇을 하는 API 인지
/**
 * GET  /api/projects — 내 프로젝트 목록 조회
 * POST /api/projects — 프로젝트 생성
 */

// ② import — 의존성
import { NextRequest } from "next/server";
import { prisma } from "@/lib/prisma";
import { requireAuth } from "@/lib/requireAuth";
import { apiSuccess, apiError } from "@/lib/apiResponse";

// ③ 메서드별 함수
export async function GET(request: NextRequest) {
  // ─── 1) 인증 ───
  const auth = await requireAuth(request);
  if (auth instanceof Response) return auth;

  // ─── 2) 입력 검증 (POST/PUT 의 경우) ───
  // body 파싱 + zod 등

  // ─── 3) 권한 체크 (필요 시) ───
  // requirePermission(...)

  // ─── 4) DB 작업 ───
  try {
    const data = await prisma.tbXxx.findMany(...);
    return apiSuccess({ items: data });
  } catch (err) {
    console.error("[GET /api/...] DB 오류:", err);
    return apiError("DB_ERROR", "조회에 실패했습니다.", 500);
  }
}
```

### 읽는 순서
1. **헤더 주석** → "이 파일이 뭐 하는 거지?" 답을 즉시 얻음
2. **함수 시그니처** (`GET`, `POST`, ...) → HTTP 메서드 확인
3. **인증 호출** (`requireAuth`) → 보호 수준 확인
4. **try 안의 비즈니스** → 핵심 로직
5. **catch** → 어떻게 실패하는가

> **20초 안에 "어떤 권한으로, 어떤 DB 를, 어떻게 조작하는지" 가 보여야 정상적인 코드.**

## 8-3. 페이지 컴포넌트 읽는 순서 — `page.tsx`

```tsx

// ⑥ 메인 페이지 export
export default function ProjectsPage() {
  return (
    <Suspense fallback={null}>
      <ProjectsPageInner />
    </Suspense>
  );
}

function ProjectsPageInner() {
  // ⑦ hook 호출 (state, query, mutation)
  // ⑧ 핸들러 함수
  // ⑨ return JSX
}
```

### 읽는 순서
1. **첫 줄 `"use client"` 여부** → 환경 결정
2. **헤더 주석의 PID-XXXXX** → 기획서 (`md/prd/UW-*.md`) 와 매핑
3. **타입** → 데이터 모양
4. **export default 함수** → 진입점
5. **그 안의 useQuery/useMutation** → 어떤 API 를 부르는지
6. **JSX 의 최상위 div** → 화면 구조
7. **이벤트 핸들러** → 동작

## 8-4. 한 파일 안의 작은 컴포넌트 패턴

[src/app/(main)/projects/page.tsx](../../src/app/(main)/projects/page.tsx) 를 보면 **한 파일에 4개의 컴포넌트** 가 있다:

```
파일 구조
├── RoleBadge              ← 5줄짜리 작은 배지 (이 파일 안에서만 씀)
├── CreateProjectDialog    ← 모달 (이 파일에서만 씀)
├── ProjectsPage           ← export default (Suspense wrapper)
└── ProjectsPageInner      ← 실제 본체
```

> 우리 컨벤션: **작은 종속 컴포넌트는 같은 파일에 둔다.** 폴더 자꾸 만들면 추적이 어려워짐.
> 두 개 이상 페이지에서 쓰면 `src/components/` 로 옮긴다.

## 8-5. import 경로 읽는 법

```ts
import { useState } from "react";                  // ← npm 패키지
import { useQuery } from "@tanstack/react-query";  // ← npm 패키지 (scoped)
import { prisma } from "@/lib/prisma";             // ← 우리 코드 (@ = src/)
import type { Theme } from "@/types/layout";       // ← 타입만 import
import "./globals.css";                            // ← CSS 파일도 import 가능
```

| 경로 | 의미 |
|---|---|
| `react`, `next` 등 | `node_modules/` 의 npm 패키지 |
| `@tanstack/react-query` | scoped 패키지 (회사/조직 prefix) |
| `@/lib/...` | `src/lib/...` ([tsconfig.json](../../tsconfig.json) 의 paths alias) |
| `./xxx`, `../xxx` | 같은 폴더, 부모 폴더 (상대 경로) |

> `@/` 만 보면 우리 코드. 그 외는 외부 의존성.

## 8-6. 화면-API-DB 흐름 트레이싱 (실전)

목록을 보여주는 화면이 있다면, **5개 지점만 따라가면** 전체가 보인다.

### 예: "내 프로젝트 목록 화면" 풀 트레이싱

| Step | 파일 | 무슨 일 |
|---|---|---|
| 1 | [src/app/(main)/projects/page.tsx:206-210](../../src/app/(main)/projects/page.tsx#L206-L210) | `useQuery` 가 `/api/projects` 호출 |
| 2 | [src/lib/authFetch.ts:60-117](../../src/lib/authFetch.ts#L60-L117) | 토큰 자동 첨부 + 만료 시 갱신 |
| 3 | [src/app/api/projects/route.ts:12-13](../../src/app/api/projects/route.ts#L12-L13) | `requireAuth` 로 인증 |
| 4 | [src/app/api/projects/route.ts:20-40](../../src/app/api/projects/route.ts#L20-L40) | Prisma 로 DB 조회 |
| 5 | [src/app/api/projects/route.ts:42-58](../../src/app/api/projects/route.ts#L42-L58) | 응답 매핑 + `apiSuccess` |

### 거꾸로 트레이싱
DB 의 `tb_pj_project_member` 테이블이 어떻게 화면까지 오는지 거꾸로 본다:

```
1. tb_pj_project_member (DB)
   ↓ Prisma 모델 = TbPjProjectMember (자동 생성)
2. prisma.tbPjProjectMember.findMany(...)   ← src/app/api/projects/route.ts:20
   ↓ apiSuccess 로 직렬화
3. JSON 응답 { data: { items: [...] } }
   ↓ authFetch
4. useQuery 의 data 에 캐싱
   ↓
5. items.map(...) 로 화면 렌더링
```

> **이 5단계가 우리 프로젝트의 모든 목록 화면에 동일하게 적용됨.**
> 새 화면 만들 때도 이 패턴 그대로 복사하면 됨.

## 8-7. 데이터 변경 흐름 (POST 의 경우)

목록과 거의 같지만 한 단계 더:

```
[브라우저] 사용자가 "생성" 버튼 클릭
          ↓
[Page]    mutation.mutate(body)                  ← src/app/(main)/projects/page.tsx:100
          ↓
[Lib]     authFetch("/api/projects", { method: "POST", body })
          ↓
[API]     export async function POST(...)        ← src/app/api/projects/route.ts:66
          │
          ├ 인증 (requireAuth)
          ├ 입력 검증 (JSON 파싱 + 필수값 체크)
          ├ 트랜잭션 시작
          │   ├ tbPjProject.create
          │   ├ tbPjProjectMember.create
          │   ├ tbPjProjectSettings.create
          │   └ tbPjProjectConfig.createMany
          └ apiSuccess({ projectId }, 201)
          ↓
[Page]    mutation.onSuccess → invalidateQueries(["projects"])
          ↓
[Page]    useQuery 가 자동 재조회 → 화면 갱신
```

이 그림이 머릿속에 들어와야 우리 프로젝트가 "한눈에" 들어온다.

## 8-11. 자주 헷갈리는 포인트

| 헷갈림 | 진실 |
|---|---|
| "API 가 안 됨" | 토큰? 권한? 둘 중 하나 거의 확실 |
| "데이터 변경했는데 화면 안 바뀜" | invalidateQueries 빠뜨림 |
| "useState 했는데 즉시 반영 안 됨" | 비동기. 다음 렌더에 반영 |
| "useEffect 무한 루프" | 의존성 배열에 매 렌더마다 새로 만들어지는 객체가 있음 |
| "params is not awaited" | Next.js 16 의 Promise params. await 필수 |
| "Hydration mismatch" | 서버/클라이언트 출력 다름. 시간/랜덤 등 의심 |
| "`use client` 가 안 먹힘" | 첫 줄에 안 옴. 반드시 첫 줄. |

## 8-12. 핵심 디버깅 도구

### 클라이언트 (브라우저)
- **Network 탭**: 어떤 API 요청이 갔고 응답이 뭔지
- **Console**: console.log 출력
- **React DevTools**: 컴포넌트 트리, props, state, hook 값
- **TanStack Query DevTools**: 캐시 상태, 쿼리 호출 이력 (개발 환경 자동)

### 서버 (터미널)
- `npm run dev` 실행 중인 터미널 → API 로그 출력
- `console.error("[GET /api/...] ...", err)` 패턴 우리 컨벤션
- Prisma 쿼리 로그: `prisma.ts` 의 log 옵션 조정

### DB
- `npm run db:studio` → Prisma Studio (브라우저 GUI)
- 직접 SQL 클라이언트로 DB 접근

## 8-14. 학습 진척도 자가 점검

이 표의 모든 항목에 "✅ 안다" 라고 답할 수 있으면 **현장 투입 가능**:

- [ ] `"use client"` 의 위치와 의미
- [ ] `params: Promise<...>` + `await params`
- [ ] `useQuery` 의 queryKey 와 queryFn
- [ ] `useMutation` + `invalidateQueries` 패턴
- [ ] `useState` 의 비동기성
- [ ] Spread (`...`) 와 Destructuring (`{ a, b }`)
- [ ] `??` 와 `?.` 의 차이
- [ ] Prisma 의 `findMany` + `where` + `include`
- [ ] `prisma.$transaction(async (tx) => {})` 의 콜백 안에서 `tx` 사용
- [ ] `requireAuth` 가 401 Response 를 반환할 때 처리법
- [ ] `apiSuccess` / `apiError` 의 응답 모양
- [ ] Zustand store 의 selector 패턴
- [ ] 우리 [.claude/develop/A-NEXTJS-기술규칙.md](../../.claude/develop/A-NEXTJS-기술규칙.md) 1~9번 규칙
