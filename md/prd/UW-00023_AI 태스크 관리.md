# UW-00023 AI 태스크 관리

> ## 1. 개요
| 항목 | 내용 |
|:-----|:-----|
| **단위업무ID** | UW-00023 |
| **단위업무명** | AI 태스크 관리 |
| **비즈니스 목적** | 영역·기능에서 발생하는 AI 요청을 태스크 단위로 관리한다. 결제 상태에 따라 자동·수동으로 AI를 호출하며, 요청 유형별 프롬프팅을 조정하여 명세 검토·설계·구현 가이드·목업·영향도 분석을 지원한다. IN_PROGRESS 상태가 5분 이상 지속되는 좀비 태스크에 대해 강제 취소를 지원하여 사용자가 무한 대기에 빠지지 않도록 한다. |
| **관련 요구사항** | RQ-00023, RQ-00021, RQ-00022 |
| **기술 스택** | AI Pipeline |

#### FID-00184 AI 태스크 재요청

#### 기능: [FID-00184] AI 태스크 재요청

| 항목 | 내용 |
|:-----|:-----|
| **기능유형** | INSERT |
| **API** | `POST /api/projects/{projectId}/ai-tasks/{taskId}/retry` |
| **트리거** | [재요청] 버튼 클릭 시 (FAILED/REJECTED/TIMEOUT 상태) |

**기술 매핑**
| 구분 | 테이블 | 주요 컬럼 |
|:-----|:-------|:----------|
| 생성 | <TABLE_SCRIPT:tb_ai_task> | prjct_id, ref_ty_code, ref_id, task_ty_code, coment_cn, req_snapshot_data, parent_task_id |

**처리 로직**
~~~
1. 재요청 API 호출 (FAILED, REJECTED, TIMEOUT 상태 대상)
2. 기존 태스크 유형, 코멘트, 스냅샷 데이터를 그대로 복사하여 새로운 tb_ai_task 레코드 INSERT
3. [v3 정책] 기존 태스크 ID를 parent_task_id에 매핑하여 이력 연결
4. 새 태스크의 retry_cnt = 3 (재시도 횟수 초기화)
5. 성공 시 상태 IN_PROGRESS로 시작 + 목록 갱신
~~~

**에러 처리**
| 상황 | HTTP | 메시지 |
|:-----|:-----|:-------|
| 재요청 불가 상태 | 400 | 현재 상태에서는 재요청할 수 없습니다 |
| 서버 오류 | 500 | 재요청 처리 중 오류가 발생했습니다 |

#### FID-00201 AI 태스크 강제 취소

#### 기능: [신규] AI 태스크 강제 취소

| 항목 | 내용 |
|:-----|:-----|
| **기능유형** | UPDATE |
| **API** | `POST /api/projects/{projectId}/ai-tasks/{taskId}/cancel` |
| **트리거** | [강제 취소] 버튼 클릭 시 (IN_PROGRESS 상태, req_dt 기준 5분 초과) |

**Input**
| 파라미터 | 타입 | 필수 | 설명 |
|:---------|:-----|:-----|:-----|
| taskId | string | Y | 취소 대상 AI 태스크 ID |

**처리 로직**
~~~
1. 확인 다이얼로그 표시: 'AI 처리가 지연되고 있습니다. 강제 취소하시겠습니까?'
2. 확인 시 강제 취소 API 호출
3. 서버 처리:
   a. tb_ai_task에서 해당 태스크 조회
   b. task_sttus_code = 'IN_PROGRESS' AND req_dt + 5분 < NOW() 검증
   c. task_sttus_code = 'FAILED'로 UPDATE
   d. result_cn = '사용자 강제 취소 (처리 지연)' 기록
   e. exec_avlbl_dt = NULL (자동 재시도 방지)
   f. AI 파이프라인에 취소 시그널 전송 (가능한 경우)
4. 성공 시 목록 갱신 + '태스크가 취소되었습니다' 토스트 표시
5. 취소 후 해당 행에 [재요청] 버튼 노출 (FAILED 상태이므로)
~~~

**참조 테이블**
| 테이블 | 컬럼 | 용도 |
|:-------|:-----|:-----|
| tb_ai_task | task_sttus_code | 'IN_PROGRESS' → 'FAILED' 변경 |
| tb_ai_task | result_cn | '사용자 강제 취소 (처리 지연)' 기록 |
| tb_ai_task | req_dt | 5분 초과 여부 검증 기준 |
| tb_ai_task | exec_avlbl_dt | NULL 세팅 (자동 재시도 방지) |

**에러 처리**
| 상황 | HTTP | 메시지 |
|:-----|:-----|:-------|
| 5분 미경과 | 400 | 아직 처리 중입니다. 잠시 후 다시 시도해 주세요 |
| 이미 완료된 태스크 | 409 | 이미 처리가 완료된 태스크입니다 |
| 서버 오류 | 500 | 취소 처리 중 오류가 발생했습니다 |

#### FID-00187 AI 결과 반영

#### 기능: [FID-00187] AI 결과 반영

| 항목 | 내용 |
|:-----|:-----|
| **기능유형** | UPDATE |
| **API** | `POST /api/projects/{projectId}/ai-tasks/{taskId}/apply` |
| **트리거** | [반영] 버튼 클릭 시 |

**기술 매핑**
| 구분 | 테이블 | 주요 컬럼 |
|:-----|:-------|:----------|
| 수정 | <TABLE_SCRIPT:tb_ai_task> | task_sttus_code('APPLIED'), apply_dt |
| 연동 | <TABLE_SCRIPT:tb_ds_function> | spec_cn 등 (요청 유형에 따라 분기 업데이트) |
| 이력 | <TABLE_SCRIPT:tb_ds_design_change> | AI 반영에 의한 설계 변경 기록 |

**처리 로직**
~~~
1. 결과 반영 API 호출
2. 서버에서 task_ty_code를 판단하여 대상 엔티티(영역의 상세설명, 기능의 명세 등)에 데이터 UPDATE
3. tb_ai_task.task_sttus_code = 'APPLIED', apply_dt = NOW()
4. tb_ds_design_change INSERT (ai_req_yn = 'Y', ai_task_id = 현재 태스크 ID)
5. 성공 시 POPUP 닫기 및 대상 상세 화면 이동 여부 확인 알림
~~~

**에러 처리**
| 상황 | HTTP | 메시지 |
|:-----|:-----|:-------|
| 이미 반영됨 | 409 | 이미 반영된 태스크입니다 |
| 서버 오류 | 500 | 반영 처리 중 오류가 발생했습니다 |
