#### FID-00126 트리 전체 로드

#### 기능: [FID-00126] 트리 전체 로드

| 항목 | 내용 |
|:-----|:-----|
| **기능유형** | SELECT |
| **API** | `GET /api/projects/{projectId}/planning/tree` |
| **트리거** | 화면 진입 시 자동 실행, 항목 CUD 성공 후 캐시 무효화 |

**Output**
| 필드 | 타입 | 설명 |
|:-----|:-----|:-----|
| tasks | array | 과업 노드 배열 (하위 요구사항 포함) |
| tasks[].taskId | string | 과업 UUID |
| tasks[].displayId | string | SFR-NNNNN 표시 ID |
| tasks[].name | string | 과업명 |
| tasks[].category | string | NEW_DEV / IMPROVE / MAINTAIN |
| tasks[].reqCount | number | 하위 요구사항 수 |
| tasks[].requirements | array | 요구사항 노드 배열 (하위 스토리 포함) |
| tasks[].requirements[].storyCount | number | 하위 스토리 수 |
| tasks[].requirements[].stories | array | 스토리 노드 배열 |
| unclassifiedReqs | array | task_id = null인 미분류 요구사항 |
| totalTaskCount | number | 전체 과업 수 |
| totalReqCount | number | 전체 요구사항 수 |
| totalStoryCount | number | 전체 스토리 수 |

**처리 로직**
~~~
1. 프로젝트 멤버십 ACTIVE 확인
2. tb_rq_task, tb_rq_requirement, tb_rq_user_story 병렬 조회 (Promise.all)
3. 메모리에서 Map으로 트리 계층 구성 (DB round trip 최소화)
4. sort_ordr 오름차순 정렬 유지
5. task_id = null인 요구사항 → unclassifiedReqs 별도 분리
~~~

**참조 테이블**
| 테이블 | 컬럼 | 용도 |
|:-------|:-----|:-----|
| tb_rq_task | task_id, task_display_id, task_nm, ctgry_code, sort_ordr | 과업 노드 |
| tb_rq_requirement | req_id, req_display_id, req_nm, priort_code, src_code, task_id, sort_ordr | 요구사항 노드 |
| tb_rq_user_story | story_id, story_display_id, story_nm, req_id, sort_ordr | 스토리 노드 |

**에러 처리**
| 상황 | HTTP | 메시지 |
|:-----|:-----|:-------|
| 프로젝트 비멤버 | 403 | 접근 권한이 없습니다 |
| DB 오류 | 500 | 기획 트리 조회에 실패했습니다 |

#### FID-00132 트리 항목 추가

#### 기능: [FID-00132] 트리 항목 추가

| 항목 | 내용 |
|:-----|:-----|
| **기능유형** | INSERT |
| **API** | 과업: `POST /api/projects/{projectId}/tasks` / 요구사항: `POST /api/projects/{projectId}/requirements` / 스토리: `POST /api/projects/{projectId}/user-stories` |
| **트리거** | 각 레벨 [+] 버튼 클릭 시 |

**Input — 과업 생성**
| 파라미터 | 타입 | 필수 | 기본값 | 설명 |
|:---------|:-----|:-----|:-------|:-----|
| name | string | Y | "새 과업" | 과업명 |
| category | string | N | "NEW_DEV" | NEW_DEV / IMPROVE / MAINTAIN |

**Input — 요구사항 생성**
| 파라미터 | 타입 | 필수 | 기본값 | 설명 |
|:---------|:-----|:-----|:-------|:-----|
| taskId | string | N | "" | 상위 과업 ID (비어 있으면 미분류) |

**Input — 스토리 생성**
| 파라미터 | 타입 | 필수 | 기본값 | 설명 |
|:---------|:-----|:-----|:-------|:-----|
| requirementId | string | Y | - | 상위 요구사항 ID |
| name | string | Y | "새 사용자스토리" | 스토리명 |
| persona | string | N | "" | 페르소나 설명 |
| scenario | string | N | "" | 시나리오 설명 |
| acceptanceCriteria | array | N | [] | 인수기준 초기값 |

**Output**
| 필드 | 타입 | 설명 |
|:-----|:-----|:-----|
| taskId / requirementId / storyId | string | 생성된 항목 UUID |

**처리 로직**
~~~
1. 선택된 부모 노드 레벨에 따라 해당 API 호출
2. display_id는 해당 프로젝트 내 해당 유형의 MAX+1 자동 채번
   - 과업: SFR-NNNNN / 요구사항: REQ-NNNNN / 스토리: STR-NNNNN
3. sort_ordr는 해당 부모 하위 마지막 + 1
4. 생성 성공 시 ["planning-tree"] 캐시 무효화 후 새 항목을 selected 상태로 설정
   → 우측 패널 자동 활성화 (편집 모드 진입)
~~~

**참조 테이블**
| 테이블 | 컬럼 | 용도 |

| tb_rq_requirement | req_id(PK), req_display_id(MAX+1), req_nm, priort_code, src_code, task_id, sort_ordr | 요구사항 생성 |

   - 요구사항 → 다른 과업: task_id 변경 + 목적지 sort_ordr 말미 배치
   - 스토리 → 다른 요구사항: req_id 변경 + 목적지 sort_ordr 말미 배치

#### FID-00129 트리 항목 삭제

#### 기능: [FID-00129] 트리 항목 삭제

| 항목 | 내용 |
|:-----|:-----|
| **기능유형** | DELETE |
| **API** | 과업: `DELETE /api/projects/{projectId}/tasks/{taskId}` / 요구사항: `DELETE /api/projects/{projectId}/requirements/{requirementId}` / 스토리: `DELETE /api/projects/{projectId}/user-stories/{storyId}` |
| **트리거** | 노드 호버 [×] 버튼 클릭 후 확인 다이얼로그에서 [삭제] 클릭 |

**처리 로직**
~~~
1. [×] 버튼 클릭 시 항목 레벨별 경고 메시지 확인 다이얼로그 표시
   - 과업: '과업과 하위 요구사항·스토리가 모두 삭제됩니다.'
   - 요구사항: '요구사항과 하위 스토리가 모두 삭제됩니다.'
   - 스토리: '사용자스토리와 인수기준이 삭제됩니다.'
2. 확인 클릭 시 해당 레벨 API 호출 (CASCADE 삭제)
3. 삭제 성공 시:
   - ["planning-tree"] 캐시 무효화
   - 삭제된 항목이 현재 selected 상태이면 selected = null (우측 패널 초기화)
~~~

**참조 테이블**
| 테이블 | 컬럼 | 용도 |
|:-------|:-----|:-----|
| tb_rq_task | task_id | 과업 삭제 (CASCADE → requirement) |
| tb_rq_requirement | req_id | 요구사항 삭제 (CASCADE → user_story → acceptance_criteria) |
| tb_rq_requirement_history | req_id | 요구사항 삭제 시 CASCADE 이력 삭제 |
| tb_rq_user_story | story_id | 스토리 삭제 (CASCADE → acceptance_criteria) |
| tb_rq_acceptance_criteria | story_id | 스토리 삭제 시 CASCADE 삭제 |

**에러 처리**
| 상황 | HTTP | 메시지 |
|:-----|:-----|:-------|
| 권한 없음 (PM/DESIGNER) | 403 | 삭제 권한이 없습니다 |
| 서버 오류 | 500 | 삭제 중 오류가 발생했습니다 |

#### FID-00130 항목 상세 조회

#### 기능: [FID-00130] 항목 상세 조회

| 항목 | 내용 |
|:-----|:-----|
| **기능유형** | SELECT |
| **API** | 과업: `GET /api/projects/{projectId}/tasks/{taskId}` / 요구사항: `GET /api/projects/{projectId}/requirements/{requirementId}` / 스토리: `GET /api/projects/{projectId}/user-stories/{storyId}` |
| **트리거** | 트리 노드 클릭 시 (selected 상태 변경 시) |

**Output — 과업**
| 필드 | 타입 | 설명 |
|:-----|:-----|:-----|
| name | string | 과업명 |
| category | string | NEW_DEV / IMPROVE / MAINTAIN |
| definition | string\|null | 과업 정의 (defn_cn) |

**Output — 요구사항**
| 필드 | 타입 | 설명 |
|:-----|:-----|:-----|
| name | string | 요구사항명 |
| priority | string | 우선순위 |
| source | string | 출처 |
| rfpPage | string\|null | RFP 페이지 번호 |
| originalContent | string | 원문 (orgnl_cn) |
| currentContent | string | 현행화 (curncy_cn) |
| analysisMemo | string | 분석 메모 (analy_cn) |
| detailSpec | string | 상세 명세 (spec_cn) |

**Output — 스토리**
| 필드 | 타입 | 설명 |
|:-----|:-----|:-----|
| requirementId | string | 상위 요구사항 ID |
| name | string | 스토리명 |
| persona | string | 페르소나 (persona_cn) |
| scenario | string | 시나리오 (scenario_cn) |
| acceptanceCriteria | array | [{acId, given, when, then}] |

**처리 로직**
~~~
1. 클릭한 노드의 type(task/requirement/story)에 따라 해당 API 호출
2. queryKey에 [타입, projectId, id] 포함 → 동일 항목 재클릭 시 캐시 재사용
3. 각 필드를 로컬 state로 바인딩하여 편집 모드로 표시
~~~

#### FID-00131 항목 저장

#### 기능: [FID-00131] 항목 저장

| 항목 | 내용 |
|:-----|:-----|
| **기능유형** | UPDATE |
| **API** | 과업: `PUT /api/projects/{projectId}/tasks/{taskId}` / 요구사항: `PUT /api/projects/{projectId}/requirements/{requirementId}` / 스토리: `PUT /api/projects/{projectId}/user-stories/{storyId}` |
| **트리거** | [저장] 버튼 클릭 시 |

**Input — 과업 수정**
| 파라미터 | 타입 | 필수 | 설명 |
|:---------|:-----|:-----|:-----|
| name | string | Y | 과업명 |
| category | string | Y | NEW_DEV / IMPROVE / MAINTAIN |
| definition | string | N | 과업 정의 |

**Input — 요구사항 수정**
| 파라미터 | 타입 | 필수 | 설명 |
|:---------|:-----|:-----|:-----|
| name | string | Y | 요구사항명 |
| priority | string | Y | HIGH / MEDIUM / LOW |
| source | string | Y | RFP / ADD / CHANGE |
| rfpPage | string | N | RFP 페이지 번호 |
| originalContent | string | N | 원문 |
| currentContent | string | N | 현행화 |
| analysisMemo | string | N | 분석 메모 |
| detailSpec | string | N | 상세 명세 |

**Input — 스토리 수정**
| 파라미터 | 타입 | 필수 | 설명 |
|:---------|:-----|:-----|:-----|
| requirementId | string | Y | 상위 요구사항 ID |
| name | string | Y | 스토리명 |
| persona | string | N | 페르소나 |
| scenario | string | N | 시나리오 |
| acceptanceCriteria | array | N | [{acId?, given, when, then}] Upsert 목록 |

**처리 로직**
~~~
1. 필수 필드 공백 검증 (명칭 비어있으면 저장 차단)
2. 해당 타입 수정 API 호출
3. 서버 처리:
   a. 과업: tb_rq_task UPDATE + mdfcn_dt = NOW()
   b. 요구사항:
      - tb_rq_requirement UPDATE + mdfcn_dt = NOW()
      - tb_rq_requirement_history INSERT 자동 이력 생성
        (vrsn_ty_code = 'INTERNAL', vrsn_no = 현재 최대 내부 버전 + 0.1)
   c. 스토리:
      - tb_rq_user_story UPDATE + mdfcn_dt = NOW()
      - tb_rq_acceptance_criteria Upsert (acId 있으면 UPDATE, 없으면 INSERT, 목록 외 기존 항목 DELETE)
4. 성공 시 '저장되었습니다' 토스트 + ["planning-tree"] 캐시 무효화 (트리 노드명 동기화)
~~~

| 테이블 | 컬럼 | 용도 |
