# 심각도 판정 기준 (모든 리뷰어 공통)

> 3개 리뷰어(PRD / Code / UI)가 **동일한 기준**으로 이슈 심각도를 매기기 위한 공통 규칙.

---

## 🔴 critical — 반드시 수정

다음 중 하나라도 해당되면 critical.

- **빌드/실행 실패를 유발**: 타입 에러, 컴파일 에러, 런타임 에러
- **PRD 요구사항의 필수 항목 누락**: 화면/영역/필수 기능/API 자체가 없음
- **보안 위험**: 인증 누락, 입력 검증 누락, 환경변수 하드코딩, SQL/XSS 취약점
- **데이터 무결성 위험**: 논리삭제 필터(`useYn: "Y"`) 누락, 트랜잭션 없는 연쇄 쓰기
- **Next.js 16 필수 패턴 위반**: `await params` 누락, `useSearchParams` Suspense 없이 사용, `"use client"` 위치 오류
- **하드코딩된 색상/크기/폰트** — 토큰 미사용 (UI 작업 절대 규칙 위반)

**판정 원칙**: critical은 "사용자가 이 상태로 쓰면 망한다"는 기준. 애매하면 major로.

---

## 🟡 major — 수정 권장

- **기술규칙 명시적 위반**: 안티패턴 테이블(any 남발, inline fetch, window.confirm, catch {}) 해당
- **PRD의 선택 기능 누락**: 에러 메시지 누락, 권한별 분기 누락, 로딩/빈 상태 처리 없음
- **디자인 일관성 붕괴**: 다른 메뉴와 패턴이 다름 (예: 삭제 확인을 ConfirmDialog 안 쓰고 직접 만듦)
- **주석 규칙 위반**: 파일 상단 역할 주석 누락, 복잡한 조건에 "왜" 설명 없음
- **파일 길이 초과**: 300줄 초과인데 분리 안 됨
- **MCP 동기화 누락**: API 인터페이스 변경했는데 MCP 도구 미수정

**판정 원칙**: "지금은 돌아가지만 유지보수에서 문제 생김"이면 major.

---

## 🟢 minor — 개선 제안

- **네이밍 개선 여지**: 상수/변수 이름이 불명확
- **추가 주석이 있으면 좋을 부분**: 뻔한 곳 말고, 미래의 자신에게 도움될 만한 부분
- **중복 로직**: 2번 복사-붙여넣기 수준 (3번 이상이면 major)
- **마이너한 디자인 편차**: spacing이 토큰을 쓰되 `--space-3`이 더 적절해 보이는 정도
- **최적화 여지**: 병렬 처리 가능한 순차 API 호출, 캐싱 부재 등 (성능 크리티컬 아니면 minor)

**판정 원칙**: "없어도 되지만 있으면 좋다"는 전부 minor.

---

## 점수 환산 (보조 지표)

최종 점수는 심각도의 **가중 감점** 방식. 절대값보다 **반복별 추세** 용도.

```
시작 점수 = 100
감점:
  critical × 15점
  major    × 5점
  minor    × 1점

최종 점수 = max(0, 100 - 감점 합)
```

예시:
- critical 1, major 3, minor 5 → 100 - 15 - 15 - 5 = 65점
- critical 0, major 2, minor 4 → 100 - 10 - 4 = 86점

## 종료 조건 (오케스트레이터 참고)

| 상태 | 조건 | 다음 행동 |
|------|------|----------|
| ✅ Pass | critical=0, major≤3 | 검토 종료 |
| ⚠️ Warn | critical=0, major>3 | 리포트 출력, 사용자 판단 |
| ❌ Fail | critical≥1 | 이슈 리스트 출력, 수정 권고 |
